## 1. 監査証跡の目的

#### 目的：過去の操作や判断を、第三者が客観的に辿れる（トレースできる）状態とする

| 観点                      | 主な目的・役割                                                    |
| ------------------------- | ----------------------------------------------------------------- |
| セキュリティ              | 不正行為の検知・抑止。インシデント発生時の原因究明。              |
| コンプライアンス          | 法令（個人情報保護法等）や認証基準（ISO/IEC 27001）への準拠証明。 |
| 説明責任 (Accountability) | 「誰が・いつ・何をしたか」を正確に記録し、正当性を証明。          |
| 運用・トラブル解消        | システム変更管理の追跡、トラブルシューティングの迅速化。          |

## 2. 監査証跡の全量マップ（コンポーネント別）

### 凡例（元のまま）

- **アクセス経路**: Web＝Webサービス経由（Cognito認証）、AWS＝AWSリソース直接（IAM認証）、External＝外部連携
- **脅威カテゴリ**: Ⅰ〜Ⅵ（上表参照）
- **通知**: ◎＝必須、○＝推奨、△＝任意、-＝不要
- **現状**: ✅ 取得中、⚠️ 一部/要確認、❌ 未対応、- 対象外

### 注釈（元のまま）

- **※1** Cognito：CloudTrail管理イベントで認証APIは取得中。詳細なログイン成功/失敗ログはCognito高度なセキュリティ機能の有効化が必要
- **※2** S3：CloudTrail管理イベントでバケット操作は取得中。オブジェクトレベル（GetObject/PutObject等）はデータイベントの有効化が必要
- **※3** DynamoDB：CloudTrail管理イベントでテーブル操作は取得中。アイテムレベル（GetItem/PutItem等）はデータイベントの有効化が必要

## この並び替え版で「表（リスク→対策→コンポーネント）」が反映されている点

- **Ⅰ（アクセス・利用制限）**は _Cognito/IAM/HENNGE_ を中心に、**不正ログイン・トークン盗難・権限昇格・BOLA/IDOR** を “実装ログ（ECS/Lambda/Next.js）” と “入口ログ（WAF/CloudFront/API GW）” の両方で追えるように再配置
- **Ⅳ（データの秘匿）**は _Secrets Manager/SSM/KMS/S3/RDS/DynamoDB_ を中心に、**「シークレット漏洩」「平文保存」「データ分類未確定」**が監査ログ設計に直結するように（＝“何を残さないか”を含めて）整理
- **Ⅴ（不正追跡・監視）**は _CloudTrail/S3(WORM)/CloudWatch/AWS Config_ を軸に、**証跡の完全性（改ざん・消去耐性）**まで含めてコンポーネント単位で追えるように整理
